git基础使用

发表于2022-12-10|更新于2023-05-12|工具运用
|阅读量:

注：本文主要参考git使用心得以及git详解并在思考后进行记录，详细请参考以上两篇文章。

常用命令：
1）、cd : 改变目录。

2）、cd . . : 回退到上一个目录，直接cd进入默认目录

3）、pwd : 显示当前所在的目录路径。

4）、ls( ll ) : 都是列出当前目录中的所有文件，只不过ll(两个ll)列出的内容更为详细。

5）、touch : 新建一个文件 如 touch index.js 就会在当前目录下新建一个index.js文件。

6）、rm : 删除一个文件, rm index.js 就会把index.js文件删除。

7）、mkdir : 新建一个目录,就是新建一个文件夹。

8）、rm -r : 删除一个文件夹, rm -r src : 删除src目录

rm -rf / : 切勿在Linux中尝试！删除电脑中全部文件！
9）、mv index.html src : mv 移动文件,其中index.html 是我们要移动的文件, src 是目标文件夹。当然, 这样写,必须保证文件和目标文件夹在同一目录下。

10）、reset : 重新初始化终端/清屏。

11）、clear : 清屏。

12）、history : 查看命令历史。

13）、help : 帮助。

14）、exit : 退出。

15）、# : 注释

git原理

在git中的输入的命令实际上跟linux是一样的

git区域结构

区域分为git本地区域与remote远端仓库

整体有四个区域

工作区（即Workspace） 工作区，就是你平时存放项目代码的地方, 就是指 我们在本地编辑代码的地方
暂存区（即Index/Stage） 暂存区，用于临时存放你的改动，事实上它只是一个文件，保存即将提交到文件列表信息, 是 为了防止误操作而建立的一个区
本地库（即Repository） 仓库区（或本地仓库），就是安全存放数据的位置，这里面有你提交到所有版本的数据。其中HEAD指向最新放入仓库的版本, 是 用于管理历史版本的区域
远程仓库（即Remote） 远程仓库，托管代码的服务器，可以简单的认为是你项目组中的一台电脑用于远程数据交换, 是 放置在远端的协作库（即github或gitee上的远程库）

本地有三个区域

git本地的三个区域确切的说应该是指：git仓库中HEAD指向的版本（状态）的意思：

（Directory：使用Git管理的一个目录，也就是一个仓库，包含我们的工作空间和Git的管理空间。）

WorkSpace：需要通过Git进行版本控制的目录和文件，这些目录和文件组成了工作空间。

（.git：存放Git管理信息的目录，初始化仓库的时候自动创建。）

Index/Stage：暂存区，或者叫待提交更新区，在提交进入repo之前，我们可以把所有的更新放在暂存区。

Local Repo（即Repository）：本地仓库，一个存放在本地的版本库；HEAD会指向当前的开发分支（branch）。

（Stash：隐藏，是一个工作状态保存栈，用于保存/恢复WorkSpace中的临时状态。）

git本地操作

git本地工作流程

git在本地工作流程一般是这样的：
１、在工作目录中添加、修改文件；
２、将需要进行版本管理的文件放入暂存区域；
３、将暂存区域的文件提交到git仓库。

因此，git管理的文件有三种状态：已修改（modified）,已暂存（staged）,已提交(committed)

文件在本地的四种状态

版本控制就是对文件的版本控制，要对文件进行修改、提交等操作，首先要知道文件当前在什么状态，不然可能会提交了现在还不想提交的文件，或者要提交的文件没提交上的情况。

Untracked: 未跟踪, 此文件在文件夹中, 但并没有加入到git库, 不参与版本控制. 通过 git add xxx(文件名) 后状态将变为Staged。

Unmodify: 文件已经入库, 未修改, 即版本库中的文件快照内容与文件夹中完全一致. 这种类型的文件有两种去处, 如果它被修改(指被用户修改内容), 而变为Modified. 如果使用 git rm --cached xxx(文件路径) 移出版本库, 或者 直接清空暂存区 rm .git/index ，通过以上两个方式操作后文件则成为Untracked文件。

Modified: 文件已修改, 仅仅是修改, 并没有进行其他的操作. 这个文件也有两个去处, 通过 git add xxx(文件名)可进入暂存staged状态, 使用 git checkout 则丢弃修改过的内容, 返回到Unmodify状态, 此处的 git checkout 即从库中取出文件, 覆盖当前修改过后的内容 !

Staged: 暂存状态. 执行 git commit -m "信息备注" 则将修改同步到本地库中, 这时本地库中的文件和工作区文件又变为一致, 文件为Unmodify状态. 执行 git reset HEAD xxx(文件名) 取消暂存, 文件状态将变为Modified。

git reset HEAD xxx(文件名) 会撤销xxx文件在Git暂存区的更改,但不撤销工作目录中此文件的修改。即文件内容不受影响,但状态会由 “Staged” 变为 “Modified”

git add （工作区 -> 暂存区）
git commit （暂存区 -> 本地库）

git在本地的版本控制

不放在本地仓库中的文件，git不进行管理
必须通过add，commit操作才能将内容提交到本地仓库

查看最近自己的操作日志

进入提交记录日志列表后，可以按 q 退出

①git log
从近到远来展示 文件提交日志
（内容较多，不易展示）

上面的命令会显示所有的 commit 记录，如果你想要显示最近几条记录，你可以通过 git log -n(n为数字, 可以随意指定) 命令完成。

②git log --online
能简洁展示日志

删除工作区中文件

rm "文件名"
之后若想在库中也删除，需依次输入add，commit命令来达成删除库中该文件

之后若想找回删除的文件，直接先找到删除前的索引，
然后使用reset --hard "索引"即可找回删除的文件

git的本地基础使用

清屏

$ clear

设置用户名和邮箱

用户名：
$ git config --global user.name "用户姓名"

邮箱：
$ git config --global user.email "用户邮箱"

初始化空文件夹

$ git init
创建完成后，可以往里面加入文件

将文件从工作区加入到暂存区

$ git add 文件名.后缀名
或者 如果需要加入很多文件或变化（为本机所有的文件或变化）
可以使用：
$ git add .

表示将刚刚进行的所有的操作全部都提交到暂存区里
（包括修改过的文件，刚刚进行过的操作等等）

然后 就会发现什么反应都没有，但是
没有反应就是最好的反应！

将已加入到暂存区的文件提交到库中

$ git commit -m "为备注，最好英文"

注意：每次提交到仓库的步骤必须为：
用git add .将刚刚修改过的文件内容和刚刚进行的操作从工作区加入到暂存区–>用git commit -m "备注"来提交到仓库

查看最近操作的文件的状态

包括以下信息：是否有文件改动后并还未提交到仓库
$ git status

使文件返回到上一个提交的版本

- -hard：三个区的版本同步改动到指定版本

$ git reset --hard HERD

表示要将当前分支重置到往前数HEAD个commit的状态

注意：如果想返回到多个版本前的话，那就多加相应次数的HEAD就可以了

【此处的 HEAD 实际为指针，指向的是 目前版本】
或者：

可使用git reset --hard xxx(任意版本的索引)来直接回退到索引对应的版本（更常用于生产环境）
（查看版本和其对应的索引可使用 git log ，git log --online 或 git reflog来查看）

但是每次进行操作过后一定要记得 push 远端仓库，才能生效，不然永远就都是在本地上操作，不会涉及到远端仓库

hard最常用，除此之外：还有以下两种方式，在此不再叙述。

- -mixed
- -soft

本地分支的概念


分支：能够让一个项目在开发主要功能(主分支master)的同时，能够复制主分支的内容以形成新的功能开发的目的，且不污染主分支。

ps：菜鸟程序员在分支上写的代码不会污染到主分支上的大佬写的代码😂

查看所有分支

git branch -v

创建主分支

git branch "分支的名称"

切换分支

git checkout "想要切换到的分支的名称"

注意:当前check到了哪个分支，使用add和commit后，添加到的文件就会上传到目前check到的分支下

创建并切换到一新分支

git checkout -b xxx(新分支名)

会产生以下效应：

  • 1.创建一个名为<新分支名>的新分支
  • 2.将HEAD指针切换到新创建的分支
  • 3.将工作目录切换到新创建的分支

执行这条命令的前提是进行前面git的本地：

  • git add xxx 操作

保证文件上传到了本地仓库中才行。

把本地仓库文件提交到远程仓库后，我们再次查看提交日志。

可以看到：

（ HEAD可以理解为一个指针，指向的是 目前的版本 ）

当我们在master分支上时【即HEAD -> master】：我们可以看得到的是master分支上的文件提交记录相关信息：

每一项显示的是一个提交对象(commit)的详细信息,包含以下字段:

  • commit xxxxxxxxxx

    这是该提交的SHA-1 hash值,用于唯一标识这个提交。

  • (HEAD -> master, origin/master, origin/HEAD)

    • HEAD -> master: 表示此提交是本地master分支的末端
    • origin/master: 表示此提交也是远程origin仓库的master分支末端
    • origin/HEAD: 表示此提交也是远程origin仓库的HEAD所指向的提交(通常为master分支末端)【远程仓库默认指向master分支】

  • Author: skym1 < xxxxxxxx @qq.com >

    这显示了该提交的作者名称和邮箱。

  • Date: Tue May 2 17:15:54 2023 +0000 这是该提交的时间,按ISO 8601标准显示,时区为UTC+0。

  • Initial commit 这是该提交的提交注释(commit message)。

    所以,每一项就包含了一个提交对象的所有基本属性,以及该提交在本地和远程仓库中的位置信息。
